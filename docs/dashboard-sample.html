The mechanism

How flaky-test remediation works here

01

Detect by re-ordering

A flaky test passes in the suite's default order but fails under a different order, because an earlier test leaks shared state. We run the suite under many random seed orderings; a test that passes sometimes and fails sometimes is flaky. Here: 9 flaky tests in 5 root-cause clusters.

02

Dispatch Devin with context

Each cluster is a GitHub issue; labeling it dispatches a Devin session with a Playbook, a Knowledge base, and a Machine Snapshot of the env. Devin reproduces the flake, bisects to the leaking predecessor, and fixes the isolation at the root.

03

Verify independently

From a fresh clone of the PR branch we re-run the targets across the known-bad seeds plus fresh seeds, scan the diff for skip/@flaky/retry/sleep, and confirm only test code changed. 46 seed-runs executed so far.

04

Route on the verdict

stabilized → ready for review. cheat / still-flaky → send the exact evidence back to Devin and retry (bounded). product bug → escalate — never mask it.
