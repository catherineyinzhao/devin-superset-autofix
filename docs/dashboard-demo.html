Devin Flaky-Test Autofix

An independent verification layer for autonomous code remediation. A scan finds issues — flaky tests, security findings, code-quality lint, vulnerable dependencies — Devin fixes them, and before any PR reaches a human this system independently re-verifies the fix and rejects cheats.

01 Why this exists

"CI is green" is not the same as "correct"

Coding agents made generating fixes cheap. They did not make trusting them cheap. The bottleneck moved: a human still has to verify every agent PR — and the signals we lean on (CI passed, the agent's own summary) are exactly the ones an agent optimizing "make CI green" can satisfy without solving the problem.

Flaky tests are the sharpest proof: a flaky test passes in CI's default order and fails under a different one, so CI-green provably lies. And the cheapest way for an agent to make one "pass" is to hide it — @pytest.mark.flaky, skip, a retry. CI goes green; the bug is now invisible and the test no longer protects you.

So this system treats "Devin opened a PR" as the start of verification, not the finish line.

And why not just a scanner? Detection is the commodity half — linters, SAST, flake-detectors and dependency scanners all emit findings (this system runs a real one). The hard, valuable half is trustworthy remediation: those findings pile into a backlog nobody clears (often masked with a # noqa — every finding here was). Pairing a scanner (the trigger) with Devin (the fix) and an independent verifier (the trust) retires the backlog instead of growing it — the value a scanner alone can never deliver.

02 The harness

A bounded recursive correction loop

The harness isn't a pipeline that ends at "PR opened" — it's a closed loop that runs until a fix is provably stable or it knowingly hands off. One Devin agent does the engineering (single-threaded, full context); the trust layer around it is deterministic — execution and static analysis, not another LLM judge that could share the agent's blind spots.

The back-edge (verdict → rounds left? → session) is the recursion: a rejected PR loops back to the same session with concrete evidence.
↻

It's closed-loop control

The validator is the sensor, the verdict is the error signal, the feedback message is the corrective action, and Devin is the actuator. The loop drives toward one setpoint: a fix that is statistically stable and not gamed.

↻

Feedback is externally grounded

Each turn carries concrete, executable evidence — the exact seed that still fails, the forbidden pattern found — not "are you sure?". Intrinsic self-correction degrades (Huang et al. 2023); correction grounded in an external signal is the regime that works.

↻

Bounded recursion, full memory

The loop is depth-capped, and because the session is single-threaded it retains the entire trace of prior attempts (per Don't Build Multi-Agents) — each round builds on the last instead of restarting cold.

↻

Three exits, never a hang

Setpoint reached → ready for review. Out of the loop's domain (a product bug) → escalate. Non-convergence (rounds exhausted or oscillation) → a circuit-breaker escalates rather than spin and burn ACU. The loop never silently gives up or loops forever.

↻

Many loops, in parallel

One loop runs per cluster, and independent clusters run concurrently (bounded for cost). Parallelism is across loops, never sub-agents inside one — throughput without conflicting decisions.

03 Context & throughput

The value is in the diagnosis — and it parallelizes

Cognition's Estimating AI Engineering Productivity finds the real effort in a session is investigation, diagnosis, and setup — not lines of code — and that context is what makes that work land. Both shape this system.

→

Context, not code volume

A flaky fix is often a two-line try/finally; the value is the diagnosis behind it. So each session is handed the context that diagnosis needs up front — the failing seeds, the suspected leaker, a Playbook, a Knowledge base of past incidents, and a Machine Snapshot of the env so it never re-pays setup.

→

Parallel, but not multi-agent

Clusters are independent, so the orchestrator runs them in parallel for throughput — bounded by a concurrency cap for cost. But each unit is still a single-threaded agent (per Don't Build Multi-Agents): parallelism across tasks, never sub-agents making conflicting decisions within one.

→

Hours, credited conservatively

Output is measured the way orgs already value engineering — in hours. That same post notes merged-PR metrics miss post-merge defects and "hours don't account for quality," so here hours are credited only on independently-verified fixes — the validator is the quality gate that a merge alone isn't.

05 Why an autonomous agent

Cheaper alternatives, judged by the same validator

An org has options short of Devin. The value of an independent bar is that it judges them all the same way. Here is what each does to the bigquery-flask-g-asyncmock cluster, and the verdict the validator returns — only the autonomous agent clears it.

Dependency bot (Renovate / Dependabot)rule-based bot cannot even attempt
Bumps package versions on a schedule. Has no concept of test order or shared state -- it cannot even see this class of bug, let alone fix it.
CI auto-retry / quarantine (Trunk-style)automation cheat detected — rejected
The industry default: rerun until green, or quarantine the test. It unblocks CI by HIDING the flake -- the bug stays, and the test no longer protects you.
Rule-based fixer scriptscript still flaky
Can rerun to detect flakiness, but has no way to diagnose which predecessor leaks state or how to isolate it -- so it produces no real fix.
One-shot code-gen (LLM, no harness)LLM, no verify loop still flaky
Can attempt a fix, but without reproduce-and-verify it usually misses the root cause -- or, prompted to make CI green, quietly hides the flake.
Devin (autonomous agent)autonomous agent stabilized
Reproduces under the bad seeds, bisects to the leaking predecessor, fixes the isolation at its root, and self-corrects when the validator pushes back.

Each cheaper tool covers only part of this column. Only an autonomous agent, held to an independent bar, fills all of it — diagnose the leak, fix the root cause, survive re-verification, self-correct, and escalate a real product bug rather than mask it.
