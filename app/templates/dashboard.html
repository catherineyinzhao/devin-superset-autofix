The mechanism

How flaky-test remediation works here

01

Detect by re-ordering

A flaky test passes in the suite's default order but fails under a different order, because an earlier test leaks shared state (a global, an app/request context, a DB row). We run the full suite under many random seed orderings (pytest-randomly); a test that passes sometimes and fails sometimes is flaky. Here: 9 flaky tests, grouped into {{ metrics.total }} root-cause clusters.

02

Dispatch Devin (with context)

Each cluster is a GitHub issue; labeling it dispatches a Devin session carrying a Playbook (the fix procedure), a Knowledge base of past incidents, and a Machine Snapshot of the env. Devin reproduces the flake under the bad seeds, bisects to the leaking predecessor, and fixes the state isolation at the root.

03

Verify independently — never trust "green"

From a fresh clone of the PR branch (not Devin's word, not one CI run) we re-run the targets across the known-bad seeds (the orderings that broke them) plus fresh seeds, scan the diff for skip/@flaky/retry/sleep, and confirm only test code changed. So far: {{ metrics.total_seeds_run }} seed-runs executed.

04

Route on the verdict

stabilized → ready for human review. cheat / still-flaky → send the exact evidence back to Devin and retry (bounded). product bug → escalate to a human — never silently mask it by editing the test.
